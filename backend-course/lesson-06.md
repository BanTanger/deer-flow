# 第 6 课：MCP 与 Tool Search 为什么能让后端可扩展

工具系统解决的是“模型怎么真正动手”，但当一个 Agent 后端开始接越来越多外部能力时，很快又会撞上另一个更隐蔽的问题：

**能力接得越多，系统越容易变重、变乱、变难维护。**

很多项目一开始会手写几套工具集成，看起来很顺。但只要外部能力继续增长，比如要接数据库、搜索、浏览器、第三方服务、团队自定义工具，很快就会出现三个连锁反应。第一，接入方式越来越碎，每一类能力都像私有协议。第二，工具初始化越来越重，系统一启动就要背很多不一定会用到的东西。第三，模型可见的工具越来越多，最后上下文和选择空间一起失控。

DeerFlow 处理这个问题的方式，不是继续堆更多私有适配，而是把 MCP 和 Tool Search 纳入同一套扩展逻辑里。

先看 MCP 这边的缓存骨架：

```python
_mcp_tools_cache: list[BaseTool] | None = None
_cache_initialized = False
_config_mtime: float | None = None

def get_cached_mcp_tools() -> list[BaseTool]:
    if _is_cache_stale():
        reset_mcp_tools_cache()

    if not _cache_initialized:
        ...  # lazy init

    return _mcp_tools_cache or []
```

这段代码本身不复杂，但它背后其实在解决一个非常真实的后端矛盾：

- **MCP 工具初始化往往不轻**
- **但 MCP 配置又不是一成不变**

如果每次请求都重新初始化，性能和延迟都会吃不消；如果一直吃缓存，又会在配置更新后出现“明明改了配置，怎么还是旧工具”的错觉。DeerFlow 用 `_config_mtime` 做失效判断，本质上是在做一个够务实的答案：

**工具缓存不是不能有，但缓存必须有失效边界。**

这个判断在多进程结构里尤其重要。DeerFlow 里 Gateway 和 LangGraph 不是同一个进程，配置可能在管理面被改掉，但真正使用工具的是运行面。如果没有一层缓存失效策略，两个进程很快就会对“当前工具世界长什么样”有不同理解。

不过，DeerFlow 真正高明的地方还不止缓存。更关键的是它没有把所有 MCP 工具的 schema 一口气全塞给模型，而是在 Tool Search 打开时，把这些能力先放进“可搜索、可延迟加载”的体系里。也就是说，模型先知道“系统外面可能有这些能力”，真正要用时再显式搜索、再加载。

这背后的设计判断非常值得学：

**能力可以很多，但模型不应该在第一轮就背下全部能力细节。**

你可以把 Tool Search 理解成一种“能力目录层”。它不是让模型什么都看不见，而是让模型先看到能力入口，再在有明确需要时去展开具体工具。这个思路和第 2 课里讲的 skills progressive loading 很像，都是同一个原则：

- **先知道有什么**
- **再按需读取细节**
- **不要一上来把所有说明书塞进上下文**

这会直接带来三个收益。

第一，prompt 和 tool binding 都会更轻。很多外部工具最重的不是调用本身，而是 schema 和说明文档。如果全部暴露，上下文会迅速膨胀。Tool Search 把这部分重量推迟到“真的要用的时候”。

第二，模型更不容易乱选工具。可选项一多，模型并不会自动变聪明，反而更容易被无关能力干扰。延迟暴露能让模型把注意力放在当前任务真正相关的那一小部分工具上。

第三，系统扩展成本更可控。因为新增一个外部能力，不再意味着“从今天起每轮上下文都更重一点”，而是“从今天起系统的能力目录里多了一个可按需加载的入口”。这两者对长期演化的影响完全不一样。

所以 MCP 在 DeerFlow 里，不只是“支持标准协议接工具”这么简单。它更像 DeerFlow 给后端扩展能力定下的一条原则：

**新增能力时，优先统一接入方式，而不是新增一套特例。**

你可以把这一层最容易踩的坑总结成三类：

- **每接一个系统就手写一套私有协议**
- **把所有外部工具 schema 一次性绑定给模型**
- **缓存有了，但没有配置变化时的失效机制**

DeerFlow 绕开的，正是这些最容易把系统拖重的地方。它用 MCP 统一扩展入口，用缓存解决初始化成本，再用 Tool Search 解决“能力很多但不该一次全给”的上下文压力。

从更大的架构视角看，这一课其实在回答一个长期问题：

**当系统能力持续增长时，怎么保证后端不会被自己的能力数量拖垮？**

DeerFlow 的答案不是“限制能力”，而是“分层暴露能力”。能力层面可以越来越丰富，但运行时可见的那一部分必须受控、按需、可回收。

如果把这一课压成一句话，我会这样说：

**MCP 解决的是能力接入方式统一，Tool Search 解决的是能力暴露方式克制。**

两层配合起来，DeerFlow 才能做到既可扩展，又不至于因为扩展本身把 prompt、binding 和运行成本一起压垮。

**这一课最后你最该记住的点**

- **MCP 解决的是后端能力的统一接入协议**
- **缓存不是重点，带失效策略的缓存才是重点**
- **Tool Search 的核心价值，是让能力按需暴露，而不是全量塞给模型**
- **真正可扩展的 Agent 后端，不是能力最多，而是能力增长后仍然可控**
