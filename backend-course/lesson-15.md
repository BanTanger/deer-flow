# 第 15 课：Guardrails、错误处理与安全边界

很多系统在能力还弱的时候，会把安全问题往后放。因为它看起来只是“一个会聊天的模型”，风险似乎还不明显。但只要系统开始具备工具调用、命令执行、文件写入、子 Agent 调度这些能力，安全就不再是以后再说的话题，而会立刻变成运行时设计的一部分。

这也是为什么 DeerFlow 在安全这件事上的思路特别值得学。它没有把“安全”抽象成一句空话，也没有试图靠一段 prompt 说“请谨慎操作”就结束，而是把安全拆成很多能在 runtime 里真正发挥作用的机制：guardrails、clarification 中断、循环检测、工具错误处理、sandbox 边界、subagent 限制。

先看 guardrail middleware 的一个核心片段：

```python
decision = self.provider.evaluate(gr)
if not decision.allow:
    return ToolMessage(
        content=f"Guardrail denied: tool '{tool_name}' was blocked ...",
        tool_call_id=tool_call_id,
        name=tool_name,
        status="error",
    )
```

这段代码最重要的地方，不是“能把请求拦下来”，而是它拦下之后不会让整条链路直接崩掉。系统返回的是一条 `ToolMessage`，并且状态是 `error`。这意味着这次拒绝并不会只存在于后端日志里，而是会成为 Agent 可以继续感知的上下文。

这个设计很有 DeerFlow 的一贯风格。它不是只关心“怎么挡住错误动作”，还关心：

**挡住之后，系统能不能继续稳定决策。**

因为在 Agent 场景里，很多安全和错误问题并不是“出现一次就结束”，而是会影响后续决策。如果拦截结果没有回到上下文，模型就很可能重复做同样的危险动作，或者误以为工具只是暂时没响应。DeerFlow 把拒绝结果显式转成 `ToolMessage`，等于是在告诉 Agent：

**这不是随机失败，这是一次有意义的边界反馈。**

Guardrail 这层还有一个很重要的策略点，就是 `fail_closed`。它意味着当 guardrail provider 自己出错时，系统可以选择更保守还是更宽松：

- **fail closed**：宁可先拦掉，不冒险放过
- **fail open**：安全子系统挂了，主请求先继续

很多人第一次看到这里，会把它理解成一个很小的开关。其实不是。它背后是一个真实的系统治理问题：

**当安全子系统本身失效时，你希望主系统默认更保守，还是默认更可用。**

这个问题没有所有场景通吃的唯一答案，所以 DeerFlow 没有替所有人拍板，而是把策略显式暴露出来。真正成熟的系统往往就是这样，不是假装所有权衡都不存在，而是把关键权衡做成明确选择。

不过，如果你只把 DeerFlow 的安全理解成 guardrails，就还是看窄了。它真正厉害的地方在于，安全边界并不是单点模块，而是一整条运行时链路共同完成的。

你把这条链串起来看，就很清楚了：

- **ClarificationMiddleware**：需求不清楚时先停，防止错误理解一路扩大
- **LoopDetectionMiddleware**：避免模型在错误路径上反复打转
- **ToolErrorHandlingMiddleware**：工具错误不直接炸链，而是变成可感知上下文
- **SandboxMiddleware**：把危险动作限制在受控执行环境里
- **SubagentLimitMiddleware**：防止子任务数量和扩散范围失控

这几层看起来分别属于不同主题，但本质上都在做同一件事：

**给 Agent 的行动能力划边界，并且在越界或失误时尽量保持系统可控。**

这也是为什么“安全”在 DeerFlow 里不是一个孤立模块，而更像贯穿 runtime 的治理思路。你可以把它理解成多层边界：

- prompt 层先告诉模型什么时候不要继续
- middleware 层在运行时真正拦截和收束
- sandbox 层限制动作落地的环境
- 错误处理层把失败转成可利用上下文

很多系统真正危险的地方，不是它会不会偶尔出错，而是它一出错就没有护栏，接着一路滑下去。DeerFlow 的设计明显不是在追求“绝不出错”，而是在追求：

**即使出现误判、失败或危险倾向，系统也不至于轻易失控。**

你可以把这一层最容易踩的坑总结成四种：

- **把安全理解成最后再加一道鉴权**
- **只有静态规则，没有运行时拦截**
- **安全系统自己出错时没有明确策略**
- **错误一发生就直接崩链，模型拿不到边界反馈**

DeerFlow 给出的答案则是：安全不是外面的一堵墙，而是运行时里一层层收束系统行为的机制。真正能长期跑的 Agent 后端，不是“从不犯错”，而是“犯错时还有边界、还有反馈、还有收敛能力”。

如果把这一课压成一句话，我会这样说：

**安全边界的关键，不是单次拦住了多少危险动作，而是系统在面对危险动作、工具失败和错误理解时，能不能持续保持受控。**

DeerFlow 值得你学的，正是这种把 guardrails、错误处理、clarification、loop detection、sandbox 统一看成运行时安全治理的方式。只有这样，Agent 的强能力才不会变成系统自己的风险放大器。

**这一课最后你最该记住的点**

- **安全不是单个模块，而是一整套运行时边界**
- **guardrails 的价值不只是拦截，还包括把边界反馈送回 Agent 上下文**
- **`fail_closed` / `fail_open` 体现的是明确的系统策略选择**
- **澄清中断、错误处理、循环检测、sandbox，本质上都在做安全治理**
