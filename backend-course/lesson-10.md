# 第 10 课：长上下文为什么会失控，DeerFlow 怎么收束

Agent 后端一旦开始跑长任务，很快就会撞上一个朴素但致命的问题：上下文越来越长。模型每轮都看到更多消息、更多工具输出、更多中间状态，最后不是成本飙升，就是注意力涣散。很多系统在这个阶段会变得很怪：它不是彻底坏掉，而是越来越慢、越来越散、越来越容易忘重点。

DeerFlow 不会假装这个问题不存在，而是正面加了一层长上下文治理。其中最关键的一层就是 `SummarizationMiddleware`，它会在上下文接近阈值时做摘要压缩。你可以把它理解成：不是把历史彻底扔掉，而是把历史重写成更短、但还能保留主线的版本。

除了 summarization，DeerFlow 还在旁边配了两个经常被忽略、但其实非常有效的收束手段：`TitleMiddleware` 和 `TodoMiddleware`。前者让线程主题更明确，后者在 plan mode 下把长任务拆成可追踪步骤。它们并不是“用户界面装饰”，而是在帮助系统把长任务重新组织成更稳定的结构。

**为什么长上下文会失控**

- **消息和工具结果会不断累积**
- **模型注意力不是无限的**
- **越长的历史，越容易让当前目标被稀释**
- **成本和延迟会一起上涨**

DeerFlow 的思路不是“永远保留完整历史”，而是“保留真正对后续还有用的结构”。这也是 Agent 后端和普通聊天后端最大的区别之一：你不是在忠实存档，而是在维护任务持续推进所需的最小上下文。

**这一课最后你最该记住的点**

- **长上下文不是资源，往往先是负担**
- **摘要不是删历史，而是重写历史**
- **Title 和 Todo 这类结构化信息，也是在帮系统收束长任务**
- **Agent 后端要维护的是“可继续推进的上下文”，不是“所有历史原文”**
