# 第 10 课：长上下文为什么会失控，DeerFlow 怎么收束

很多人做 Agent，前期最容易沉迷的是“上下文终于够长了”。因为在普通聊天时代，大家最怕的是模型忘事；一旦上下文窗口变大，系统看起来就像突然拥有了更强的持续任务能力。可真把长任务跑起来以后，你会发现一个很反直觉的事实：

**上下文越长，不代表系统越稳，很多时候反而更容易散。**

原因很简单。Agent 后端里的上下文，不只是用户和模型的聊天记录，还包括工具调用结果、中间分析、计划调整、错误回传、文件摘要、搜索结果、子任务汇总。它们会一轮一轮叠上去。到了某个临界点，系统虽然“还装得下”，但模型已经开始抓不住重点了。于是你会看到一种很典型的失控方式：它不是直接报错，而是越来越慢、越来越贵、越来越爱重复、越来越容易偏离主线。

DeerFlow 对这个问题的判断很清醒：长上下文从来不是免费资源，它首先是一种治理对象。

所以它会在 runtime 里加入 `SummarizationMiddleware`，当上下文接近阈值时，把历史压缩成更短、但还保留任务主线的信息。这里的关键不是“删掉旧消息”，而是“重写旧消息”。

很多人第一次听到摘要，会以为这就是另一种丢历史。其实不是。真正的问题从来不是历史要不要保留，而是：

**后续步骤到底需要历史的原文，还是需要历史的结构。**

对于大多数长任务来说，后续轮次通常不需要回看每一句原话，它需要的是这些更高阶的信息：

- 已经做到了哪一步
- 哪些约束已经确认
- 哪些错误已经发生过
- 当前主目标和次目标分别是什么
- 后续还剩什么待办

这也是为什么摘要在 Agent 后端里不是压缩存档，而是重建“可继续推进的上下文”。DeerFlow 的总结思路非常像在做运行时收束：把原本散落在长对话里的行动线索、计划状态、关键结论重新折叠成一个更短、更有结构的版本。

但 DeerFlow 并没有把“控制长上下文”这件事全压给摘要。它旁边还配了两个很多人容易低估的中间件：`TitleMiddleware` 和 `TodoMiddleware`。

这两个名字看起来不如 summarization 那么“技术味”，但它们其实都在做同一件事：帮助系统在长任务里持续记住“我们到底在干什么”。

`TitleMiddleware` 的作用，不只是给线程起个好看的标题。它是在给长任务补一个主题锚点。线程一长，模型最容易丢的不是局部信息，而是全局主题。标题越清晰，系统越容易把当前动作重新挂回总任务。

`TodoMiddleware` 更不用说。它不是 UI 装饰，而是让 plan mode 下的任务步骤有了显式状态。待办一旦明确，系统对“哪些已经完成、哪些还在进行、哪些尚未开始”的把握就会稳定很多。很多长任务后期之所以会乱，不是因为模型真的看不懂历史，而是因为历史里缺少足够强的结构锚点。

所以 DeerFlow 在这里的思路不是单点修补，而是一整套收束策略：

- **上下文长了，就摘要**
- **任务散了，就用标题重新挂主题**
- **步骤多了，就用 todo 维持推进结构**

这说明它把“长上下文问题”理解成一个多层问题，而不是单纯的 token 问题。真正让任务失控的，往往不是窗口不够大，而是窗口里缺少结构。

你可以把长上下文最容易坏掉的方式总结成四种：

- **什么都保留，结果什么都一样重要**
- **工具输出越积越多，但没有重新组织**
- **当前目标被历史噪声稀释**
- **系统保留了原文，却丢了推进任务的结构**

DeerFlow 避开这些坑的方式，其实和前面几课一脉相承。第 2 课讲 prompt 不要大而全，第 8 课讲 memory 要提炼而不是回灌原文，这一课继续把同一个原则贯彻到运行时：

**系统不需要永远记住所有原文，它需要永远记住任务主线。**

这句话很重要，因为它点出了 Agent 后端和普通聊天后端的差别。普通聊天可以偏向“保留全部对话”，因为目标往往是回答得像在连续聊天；而 Agent 后端的核心目标是持续推进任务，所以它必须优先维护“还有什么要做、当前卡在哪、之前已经确认了什么”这种行动结构。

你也可以重新理解“上下文长度”这个指标。很多人会把它当成能力上限，但在真实系统里，它更像预算。预算大，当然是好事；但预算大不代表就可以无纪律地往里面堆内容。预算越大，越考验系统会不会自己做结构化收束。

如果把这一课压成一句话，我会这样说：

**长上下文最大的问题不是装不下，而是装得下之后，系统还能不能继续抓住主线。**

DeerFlow 的做法值得学，正是因为它没有把上下文治理当成“快满了再删一点”，而是把它当成一套持续收束主线的运行时机制。摘要、标题、todo，看起来分别作用在不同位置，本质上却都在做同一件事：让长任务不要越跑越散。

**这一课最后你最该记住的点**

- **长上下文不是天然优势，很多时候先是治理负担**
- **摘要的关键不是删历史，而是重写出更可推进的历史结构**
- **Title 和 Todo 不是装饰，它们也是任务收束手段**
- **Agent 后端真正要维护的，是“还能继续推进的上下文”，不是“所有历史原文”**
