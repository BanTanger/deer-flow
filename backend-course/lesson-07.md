# 第 7 课：Sandbox 为什么不是可选项，而是底线

很多人第一次给 Agent 接 bash、文件读写或者代码执行时，第一反应通常是兴奋：它终于不像聊天机器人了，它真的能“动起来”了。可如果你从后端视角看，恰恰是从这一刻开始，系统的风险等级会明显上升。

因为一旦 Agent 不只是生成文本，而是能执行命令、修改文件、处理上传内容、调用外部环境，你就不再是在设计一个“回答系统”，而是在设计一个“会行动的系统”。会行动，就意味着必须考虑边界；会落盘，就意味着必须考虑隔离；会跨轮继续任务，就意味着必须考虑生命周期和清理策略。

DeerFlow 对这件事的态度很明确：执行环境不能靠模型自己理解，也不能默认把宿主机直接暴露给模型。它先用 `SandboxMiddleware` 把当前线程绑定到一个明确的 sandbox，再允许后续动作发生：

```python
if "sandbox" not in state or state["sandbox"] is None:
    thread_id = runtime.context["thread_id"]
    sandbox_id = self._acquire_sandbox(thread_id)
    return {"sandbox": {"sandbox_id": sandbox_id}}
```

这段代码看起来很短，但它其实先做了一个特别重要的决定：

**sandbox 的归属是 thread 级，不是全局级。**

这个判断为什么关键？因为如果多个线程共享同一份执行环境，很多问题会立刻冒出来：文件会相互污染，命令副作用会串线，产物归属会模糊，清理时机也会越来越难判断。Agent 系统一旦允许“执行”，最怕的往往不是单次命令失败，而是执行现场混在一起。

所以 DeerFlow 不是简单造一个容器或临时目录就结束，而是把 sandbox 和线程强绑定。这意味着用户每开一条线程，系统都会尽量给它准备一个可追踪、可管理、可回收的独立执行空间。

但光有 sandbox ID 还不够。DeerFlow 还会配合 `ThreadDataMiddleware`，把 workspace、uploads、outputs 这些路径先放进状态里。这样执行环境就不再只是“某个容器”或者“某个目录”，而是一整套线程级任务现场：

- **上传内容放在哪里**
- **临时工作在哪进行**
- **最终产物应该落到哪里**
- **后续工具应该基于哪份路径上下文行动**

这件事特别重要，因为很多文件类错误并不是“命令执行失败”，而是系统在错误位置成功执行了错误动作。比如把临时处理结果误当最终交付物，或者在不该写的目录里写了内容。DeerFlow 通过 thread_data + sandbox 这套组合，先把舞台搭清楚，再让模型开始行动。

你可以把 sandbox 理解成 DeerFlow 在执行层设下的一条底线：

**模型可以行动，但行动必须发生在受控环境里。**

这也是为什么“有没有 sandbox”不是一个体验增强项，而是一个系统分水岭。没有 sandbox 的早期 demo 往往也能跑，但只要任务复杂度一上来，就会迅速暴露出这些问题：

- **Agent 默认依赖宿主环境细节**
- **线程之间的执行痕迹混在一起**
- **文件写入和命令执行没有明确边界**
- **资源回收时不知道该清什么，也不知道什么时候清**

很多人会说：“我只是本地开发，真的需要这么正式吗？” 这个问题看似合理，但其实恰好暴露出一个常见误区。Sandbox 的价值不只是安全，还是一致性。它让你的系统在开发、调试、生产里都尽量基于同一套执行假设工作，而不是每次都去赌当前宿主环境恰好满足模型的隐含预期。

从这个角度看，sandbox 解决的不止一个问题：

- **安全边界问题**
- **线程隔离问题**
- **环境一致性问题**
- **生命周期管理问题**

DeerFlow 的做法其实非常有“后端感”。它不是把执行能力直接交给模型，然后靠 prompt 告诉模型“请小心一点”；它是在 runtime 先把环境装起来，让后续所有动作都自然落在这个边界里。也就是说，安全不是靠模型自觉，而是靠环境先行约束。

你也能看出第 7 课和前面几课之间的呼应。第 4 课讲 ThreadState 是状态背板，第 5 课讲工具层让模型能做动作，到这一课其实就在补一句非常关键的话：

**会做动作之后，马上就要开始约束动作。**

如果没有这一层，前面所有“能力增强”最后都会变成新的风险来源。工具越强，越该先问清楚：动作在什么边界里发生，谁来负责隔离，谁来负责清理。

如果把这一课压成一句话，我会这样说：

**Sandbox 的核心意义，不是“给 Agent 一个能跑命令的地方”，而是“给 Agent 一个受控、隔离、可管理的执行现场”。**

只有这层底线先立住了，bash、文件处理、代码生成、artifact 输出这些能力才不是系统负担，而是真正可持续的后端能力。

**这一课最后你最该记住的点**

- **执行能力一打开，隔离就不再是可选项**
- **DeerFlow 的 sandbox 是 thread 级资源，不是全局共享环境**
- **sandbox 和 thread_data 组合起来，才构成完整的执行现场**
- **没有受控执行边界的 Agent 后端，能力越强，后面越难收场**
