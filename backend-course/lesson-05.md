# 第 5 课：工具系统是怎么把“会说”变成“能做”的

很多人学 Agent，最容易高估的是 prompt，最容易低估的是工具层。因为 prompt 很显眼，看起来像“智能”的来源；工具层却常常被误解成“接几个 API 而已”。但如果你从后端能力角度看，真正决定 Agent 能不能落地执行的，往往不是 prompt，而是工具系统。

原因很简单。模型天然只会输出文本。它可以说“我建议读取这个文件”，可以说“我建议搜索一下”，也可以说“我现在应该执行命令”。可如果后端没有把这些动作正式接成工具，模型说得再像，也只是描述一个本来应该发生、但并不会真的发生的动作。

DeerFlow 的工具系统，解决的就是这条鸿沟：

```python
def get_available_tools(
    groups: list[str] | None = None,
    include_mcp: bool = True,
    model_name: str | None = None,
    subagent_enabled: bool = False,
) -> list[BaseTool]:
```

这里最关键的一点是，函数返回的不是“工具配置说明”，而是 `BaseTool` 列表，也就是一批真正可以被 runtime 绑定、被模型调用、被系统执行的工具对象。换句话说，DeerFlow 不是在告诉模型“理论上你会这些”，而是在把这些能力正式接进执行回路。

再往里看，核心逻辑其实很能说明 DeerFlow 的后端思路：

```python
loaded_tools = [
    resolve_variable(tool.use, BaseTool)
    for tool in config.tools
    if groups is None or tool.group in groups
]

builtin_tools = BUILTIN_TOOLS.copy()

if subagent_enabled:
    builtin_tools.extend(SUBAGENT_TOOLS)

if model_config is not None and model_config.supports_vision:
    builtin_tools.append(view_image_tool)
```

很多人第一次看这段代码，会觉得它只是在“拼一个列表”。但 DeerFlow 真正厉害的地方，不是拼接动作本身，而是它清楚知道：

**工具不是越多越好，而是当前这次运行该给哪些。**

这是 Agent 后端特别容易踩的一个坑。早期做系统时，大家很容易产生一种冲动：既然辛辛苦苦接了这么多工具，那就全部暴露给模型吧。看起来像是能力变强了，实际上常常是在把模型推进更大的选择混乱里。工具越多，上下文越重；工具越杂，误调概率越高；能力边界越不清，模型越容易在不该动手的时候瞎动手。

DeerFlow 的做法明显更克制，也更工程化。比如只有视觉模型才拿到 `view_image_tool`，因为不给视觉模型这个工具，模型根本没有处理图片的能力前提；只有允许 subagent 的模式才会挂上 `task` 相关工具，因为不是每一轮任务都该鼓励模型去拆分子 Agent。

这说明 DeerFlow 把工具系统看成“能力装配层”，不是“能力仓库展示层”。能力当然可以很多，但真正进入当前运行的能力集合，必须和模型能力、运行模式、系统边界一起决定。

这里还有一个特别值得你记住的点，就是 `resolve_variable(...)`。它的意义不是简单 import，而是把配置里的路径解析成真正的 Python 对象。你可以把它理解成 DeerFlow 在说：

**工具接入不应该靠手写硬编码分支，而应该靠配置 + 反射来扩展。**

这一点非常关键。因为工具系统一旦要长期演进，就一定会遇到这些现实需求：

- 团队想新增自定义工具
- 不同部署环境想启用不同工具组
- 社区工具和内建工具要统一挂载
- 有些工具要延迟加载，有些要开箱即用

如果所有工具都靠代码里手写 if/else 拼接，系统很快会越来越硬。DeerFlow 通过配置化装配，给后面的 MCP、skills、社区扩展都留出了空间。

不过，光能“注册更多工具”还不够。工具系统真正难的，是决定什么时候不要给。你可以把 Agent 工具层最容易坏掉的方式总结成四种：

- **把所有工具都暴露给模型**
- **把模型能力和工具能力混为一谈**
- **只管注册，不管错误和约束**
- **把扩展能力写死在代码里**

DeerFlow 其实是在一开始就绕开这些坑。它让工具集合动态装配，让能力暴露跟当前模型特性绑定，也让后续工具扩展不必回头重写核心 runtime。

从系统设计角度看，工具层和 prompt 层其实分工很清楚。prompt 负责让模型知道“什么时候该做什么”，工具层负责把“该做的动作”变成真正能执行的调用。前者偏软规则，后者偏硬执行。只有两层都在，Agent 才不会停留在“很会描述动作”的阶段。

你也可以把 DeerFlow 的工具系统读成一个更底层的判断：

**模型负责决策，工具负责落地，后端负责把两者可靠接起来。**

这个判断很朴素，但很多系统正是因为没把这三层分开，才会在后面越来越乱。模型说了很多漂亮话，但执行链路不完整；工具接了很多，但暴露方式混乱；后端想收敛，却没有一层统一的工具装配逻辑。

如果要把这一课压成一句话，我会这样说：

**工具系统真正解决的，不是“给模型更多按钮”，而是把模型的意图变成受控、可配置、可执行的后端动作。**

所以 DeerFlow 的工具层值得学，不是因为它接了多少工具，而是因为它清楚知道：能力暴露要看场景，能力扩展要有抽象，能力执行要和 runtime 严密接上。

**这一课最后你最该记住的点**

- **工具层是 Agent 从“会说”变成“能做”的桥**
- **工具集合不该全量暴露，而该按当前模式动态装配**
- **`resolve_variable(...)` 代表的是配置化、可扩展的工具接入方式**
- **真正稳定的工具系统，重点不在“有多少工具”，而在“哪些工具此刻该出现”**
